# ADR-005: 文本加载统一走 Worker（禁止主线程同步加载）

<!-- 状态: accepted | 最后验证: 2026-06-04 -->

## 背景

本地文本来源（如 `builtin_demo`）有 `local_path`，旧代码的 `_load_from_local()` 除了读文件外，还调用 `_lookup_server_text_id()` 回查服务端获取 `text_id`。这个回查涉及同步 HTTP 请求，可能耗时数百毫秒。当默认来源是本地来源时，每次进入跟打页面都会在主线程触发同步网络请求，导致 UI 冻结。

## 选项

### A. 本地来源同步加载，远程异步加载

根据来源类型选择加载方式。

**优点**：本地文件理论上很快。
**缺点**：本地加载隐含网络调用（回查 ID）；无法统一测试策略。

### B. 所有加载统一走 Worker

无论本地还是远程，所有文本加载统一走后台 Worker 线程。

**优点**：
- 统一加载路径，代码简洁
- 主线程永不阻塞
- 可以统一错误处理和进度反馈

**缺点**：本地小文件有 Worker 调度开销（可忽略）。

### C. 异步读取本地文件 + 异步回查

本地文件用异步 I/O，回查也异步。

**优点**：更精细的控制。
**缺点**：Linux 文件 I/O 异步 API 复杂且跨平台一致性差；与远程加载的两阶段模式不一致。

## 决策

**选择 B**。所有文本加载统一走 `TextLoadWorker`。两阶段异步：

1. **Worker 阶段**：只读文件，立即返回 `text_id=None`，载文即时显示
2. **Daemon thread 阶段**：检测到 `text_id=None` 时，启动后台 daemon thread 异步回查；成功后通过 `localTextIdResolved` 信号 → `Bridge.setTextId()` 更新

## 影响

- **正向**：UI 永不因载文阻塞
- **正向**：加载路径统一，测试覆盖简化
- **变更**：所有文本加载入口统一走 `requestLoadText()` + `_load_async()`
- **注意**：本地文本的 `localTextIdResolved` 信号是可选的（用户可能不需要排行榜关联）
